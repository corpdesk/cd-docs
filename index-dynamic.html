Introduction

Corpdesk represents a cutting-edge framework for building scalable, modular, and highly automated applications. With its unique design philosophy, Corpdesk ensures consistency, simplicity, and flexibility, enabling both human developers and AI tools to interact seamlessly with its architecture. This document outlines the foundational principles, operational modes, and core components of the Corpdesk backend, focusing on its API execution strategy.

Design Philosophy
  • Automation-First Approach: Corpdesk is designed to maximize automation while minimizing dependencies.
  • Language-Agnostic Implementation: Designs can be transformed into multiple programming languages.
  • Consistency and Simplicity: Centralized API interactions promote maintainability.
  • Uniform Entry Points: The API uses a single entry point with all requests formatted as JSON POST data.
Operational Modes
  • No Client-Side Routes: All calls are sent to a single endpoint.
  • Standardized API Calls: Requests are structured using standardized interfaces.
  • Service Initialization: Services use new() without dependency injection.
  • Environment-Aware Execution: Configurations for CORS, SSL, and database connections.
Core Interfaces
ICdRequest

The ICdRequest interface standardizes how requests are structured:

  • ctx: Specifies the target module's location.
  • m: Module name.
  • c: Controller name.
  • a: Action to execute within the controller.
  • dat: Payload containing command settings.

    // JSON request example:
    {
        "ctx": "Sys",
        "m": "User",
        "c": "User",
        "a": "Login",
        "dat": {
            "f_vals": [
                {
                    "data": {
                        "userName": "user-name",
                        "password": "*******"
                    }
                }
            ],
            "token": null
        },
        "args": null
    }
ICdResponse

The ICdResponse interface ensures consistency in responses:

  • app_state: Contains success status, error messages, and session data.
  • data: Holds the actual response payload.
Backend Workflow
  • Initialization: The entry point launches the application and executes runtime configurations.
  • Request Handling: Requests are processed, validated, executed, and responded to with structured outputs.
