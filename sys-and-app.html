Understanding the "sys" and "app" Directories

In Corpdesk, there are two fundamental directories for modules: sys and app. While all modules share the same operational framework and interact with the BaseService similarly, these directories serve distinct purposes to streamline development and reuse.

The "sys" Directory

The sys directory contains modules designed to provide foundational, reusable services that are "most likely to be utilized by other applications and application developers." These modules are typically maintained by Corpdesk's core system developers and address common needs across projects.

Modules in the sys directory target areas such as:

  • Database connections and entity management.
  • User, group, and privilege subsystems.
  • Scheduling services and time-based interfaces.
  • Communication services like email, SMS, and real-time messaging.
  • Collaboration features inspired by modern social media capabilities.

Below is the structure of the sys directory, showing its modules: 


            $> tree -L 2 src/CdApi/sys/
            src/CdApi/sys/
            ├── base
            ├── cd-push
            ├── comm
            ├── inte-ract
            ├── moduleman
            ├── scheduler
            ├── user
            └── utils
Examples of "sys" Modules
Module Description
cd-push Provides push services using WebSocket tools like socket.io.
comm Supports messaging and notifications, including email, SMS, and USSD.
inte-ract Enables collaboration features with modern social media functionalities.
moduleman Manages system-level entities such as modules, users, and resources.
scheduler Provides time-based scheduling services that other modules can depend on.
user Handles user and group management, offering ready-made services for authentication and authorization.
The "app" Directory

The app directory is designed for modules specific to an application's requirements. These modules:

  • Can be created by any developer for project-specific needs.
  • May rely on sys modules for shared services, such as email notifications or scheduling.
  • Are less likely to be reused across projects compared to sys modules.
Interdependencies Between "app" and "sys" Modules

While app modules can depend on one another, such dependencies are not categorized as "system-level" dependencies. For example:

  • cd-hrm (an HR module) may depend on cd-accts (an accounting module) for payroll processing.
  • Both modules may rely on cd-comm (a sys module) for email notifications.
Conclusion

By categorizing modules into sys and app directories, Corpdesk ensures a modular, reusable, and scalable architecture. Developers are encouraged to leverage sys modules for common needs while building app modules tailored to specific project requirements. This approach promotes consistency, efficiency, and community-driven development.
